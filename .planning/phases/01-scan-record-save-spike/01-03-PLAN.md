<objective>
Empirically resolve — on the real target Windows machine with real webcam and USB keyboard-wedge scanner hardware — the three unknowns this entire spike phase exists to answer, plus the two forward-looking data points. Per the roadmap, validating the camera pipeline against real hardware is the literal reason Phase 1 exists; it cannot be claimed complete from Linux compilation alone.

This plan is ALL human-gated checkpoints (autonomous: false) because none of it can be automated from the Linux dev environment: it requires physical webcam + scanner hardware on a Windows 10/11 machine. The Dart code under test was built in Plans 01 and 02.

Unknowns to resolve (from RESEARCH.md Assumptions A1/A2/A3 and Pitfalls 1/2/3/5):
1. `stopVideoRecording()` -> `startVideoRecording()` back-to-back reliability & latency (REC-02, the project's Core Value)
2. `HardwareKeyboard` focus-independence on 3.41.6, incl. after TextField focus (BAR-01, issue #160786)
3. `availableCameras()` enumeration of the target webcam by name (SET-01)
Plus: release-build DLL smoke test (Pitfall 5) and ffprobe codec capture (Pitfall 4, for Phase 3).

Purpose: Prove the highest-risk technical unknown before any further project work depends on it, and record findings for Phase 2-3 planning.
Output: `SPIKE_RESULTS.md` with pass/fail + measurements for every checkpoint; the `camera_desktop` fallback decision if `camera_windows` fails.
</objective>
