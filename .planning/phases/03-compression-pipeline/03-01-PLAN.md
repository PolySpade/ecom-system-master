<objective>
Port ecom-py's background compression pipeline (`ecom-py/video_compressor.py`) to the Flutter app as a service-layer feature: FFmpeg auto-discovery (COMP-02), a serialized FIFO background worker (COMP-01), crash-safe atomic file replacement (COMP-03), configurable FFmpeg invocation with Windows process priority, hidden-console intent, capped threads, and a 1-hour kill timeout (COMP-04/SET-06), and per-transaction status persistence using ecom-py's exact schema (COMP-05).

FFmpeg is NOT installed on this machine, so every behavior is verified through unit tests with injectable seams (ProcessStarter, FfmpegFinder, PriorityApplier, job timeout) rather than a live FFmpeg run. The graceful-skip path (the one that WILL trigger on this machine) is tested explicitly.

Phase 2 (concurrent worktree) owns the main screen; this plan does NOT touch lib/ui/main_screen.dart or lib/main.dart. UI exposure is limited to a self-contained CompressionStatusIndicator widget plus a ValueNotifier on the compressor that Phase 2 can slot in at merge time.

Output: five service/test units committed in small conventional commits, all `flutter analyze` clean and `flutter test` green.
</objective>

  <behavior>
    - queueCompression: disabled -> skipped callback + false (no process); FFmpeg missing -> logged warning + skipped callback + false; missing input -> failed callback + false; else enqueue + true
    - Jobs process strictly one at a time, FIFO; a second job never starts before the first completes
    - FFmpeg args: -i input -c:v libx264|libx265 -crf <clamped 18-35> -preset <preset> -threads 2 -c:a aac -b:a 128k -y <base>_compressed<ext>
    - Success requires exit code 0 AND output exists AND output size > 0
    - deleteOriginal=true: rename original -> .bak, rename temp -> original, delete .bak (no crash point leaves zero copies); deleteOriginal=false: keep both (ecom-py parity), compressed_filename = temp basename
    - Failure/timeout: partial output deleted, original untouched, 'failed' callback; timeout kills the process after jobTimeout (default 1h)
    - Priority applied post-spawn via kernel32 OpenProcess/SetPriorityClass FFI (low=IDLE, below_normal, normal)
    - queueStatus ValueNotifier reflects queueSize/isProcessing/currentTransactionId/workerRunning transitions
    - Callback resultData keys match ecom-py: status, compressed_file_size_mb, compression_ratio, compressed_filename, message

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Video path flows into an FFmpeg process argument | mitigate | Process.start with an argument LIST (no shell), runInShell=false - paths are never string-interpolated into a shell command |
| T-03-02 | Denial of Service | Runaway/hung FFmpeg process | mitigate | 1-hour timeout with forced kill (COMP-04); -threads 2 cap; configurable low process priority |
| T-03-03 | Tampering | Crash mid-compression corrupts/loses the original video | mitigate | Temp-in-same-dir + backup-rename swap: original -> .bak, temp -> original, delete .bak; every crash point leaves >=1 intact copy |
| T-03-04 | Elevation | kernel32 FFI (OpenProcess/SetPriorityClass) | accept | Only lowers priority of a child process we spawned; PROCESS_SET_INFORMATION access only; failures logged and ignored |
| T-03-05 | Spoofing | A planted ffmpeg.exe in a searched directory executes | accept | Same trust model as ecom-py (identical search order); bundled dir beside the exe is checked first, which is install-controlled |
</threat_model>
